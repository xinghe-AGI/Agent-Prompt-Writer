# Prompt Method

本文件是 AI Prompt Writer 的核心方法。不要再按很多框架拆文件；先判断 prompt 的用途，再选最小可用结构。

## 1. 先判断用途

| 用途 | 写法 |
|---|---|
| 一次性任务 | 短 prompt，保留角色、任务、约束、输出格式 |
| 结构化输出 | 使用完整骨架，重点写字段契约 |
| JSON 抽取 | 强制唯一输出形态、字段类型、缺失值、枚举和来源约束 |
| Agent 执行 | 写清目标、范围、允许动作、禁止动作、检查点、停止条件 |
| 自动化工作流 | 写清触发、步骤、字段映射、失败处理、最终产物 |
| 长期角色 | 加入简介、版本、初始化、交互规则和维护纪律 |

如果用途不清楚，最多问 3 个问题。优先问会影响输出契约的问题，例如目标工具、输出形态、字段、异常输入。

澄清问题优先级：

1. prompt 要给哪个工具或模型使用？
2. 最终输出形态是什么？
3. 字段、范围、异常输入或完成标准是否已经确定？

不要把澄清变成长问卷。能合理假设的就先说明假设，再生成 prompt。

## 2. 再判断意图

写 prompt 前先判断用户需求属于哪一类。意图判断只用于选择写法，不要把框架名暴露给用户，除非用户要求学习框架。

| 意图 | 典型情况 | 处理方式 |
|---|---|---|
| 澄清 | 用户目标大致清楚，但关键字段、输出形态或工具不清楚 | 最多问 3 个关键问题 |
| 生成 | 用户从零要一段 prompt | 选择最小结构，直接生成可复制 prompt |
| 转换 | 用户已有 prompt，想优化、改写、适配 | 保留原意，收紧任务、边界、输出格式和完成标准 |
| 评审 | 用户想知道 prompt 为什么不好 | 先指出主要问题，再给修正版 |
| 执行 | prompt 用于 Agent、编程工具或自动化工作流 | 必须写清范围、禁止动作、检查点和停止条件 |

只有当某个结构能补上“用户需求”和“理想输出”之间的缺口时，才使用它。如果需求已经清楚，不要为了显得专业而套复杂框架。

## 3. 统一核心骨架

生产级 prompt 使用这个骨架：

```text
# 角色 : <助手名称>

## 背景 :
说明输入来源、使用场景、模型易错点。

## 简介 :
- author:
- version:
- language:
- description: <输入 + 输出 + 本版关键变化>

## 任务 :
分点说明模型要完成的动作。

## 约束条件（重要）:
- 输出形态
- 缺失值策略
- 字段来源约束
- 枚举和类型约束
- 异常输入处理

## 技能 :
说明模型需要执行的识别、判断、归一化、分类能力。

## 工作流 :
1. ...
2. ...
3. ...

## 输出格式（必遵）:
定义字段、类型、枚举、顺序。

## 输入模板 :
告诉用户如何提供输入。

## 输出样例 :
至少覆盖成功样例和异常样例。

## 初始化 :
简短说明如何开始。
```

简单任务不要强行套完整骨架。压缩为：

```text
角色：
任务：
输入：
约束：
输出：
完成标准：
```

## 4. 9 条硬标准

1. **简介必须可追踪版本变化**：`description` 写清输入、输出、关键字段和本版变化。
2. **输出形态只能有一个**：JSON、Markdown、自然语言、表格、代码补丁只能选一种主形态。
3. **字段契约要全量闭环**：简介、约束、输出格式、样例、初始化中的字段口径要一致。
4. **缺失值策略要统一且可执行**：无法识别时优先返回 `null`，不要编造默认值。
5. **枚举值要可列举、可校验**：分类字段必须给允许值集合。
6. **字段来源约束是防幻觉关键**：写清哪些字段必须来自输入，哪些字段允许模型判断。
7. **规则写法用“条件 + 结果”**：少写“请准确”，多写“若 X，则输出 Y”。
8. **样例要覆盖主流程和异常流程**：至少有成功样例；生产级任务补部分缺失和非目标输入样例。
9. **初始化要短，但必须能直接开工**：重申身份、输出结构和用户输入方式即可。

## 5. 辅助原则

论文 26 条原则只作为轻量检查，不机械套用。默认采用这些稳定原则：

- 明确受众。
- 使用正向指令。
- 用标题、分隔符或代码块分开指令、输入、示例和输出。
- 示例比抽象规则更能锁定格式。
- 复杂任务拆步骤。
- 需要澄清时先提问。
- 输出格式必须具体。
- 不必要的客套、激励和惩罚语句默认不用。
- 每句话都必须改变预期输出；不改变输出的句子删掉。

## 6. 坏 Prompt 检查

交付前检查 8 个高频问题：

| 问题 | 修复方式 |
|---|---|
| 任务动词模糊 | 改成明确动作，例如抽取、分类、改写、生成、检查 |
| 多个任务混在一起 | 拆成步骤，或说明先做什么、后做什么 |
| 缺少输出格式 | 指定 JSON、Markdown、表格、代码补丁或自然语言 |
| 缺少成功标准 | 增加“完成标准”或“Done when” |
| 输入边界不清 | 用标题、分隔符或代码块包住输入 |
| 默认模型知道上下文 | 增加简短背景或记忆块 |
| Agent 缺少范围和停止条件 | 补充允许范围、禁止动作、检查点、停止并询问 |
| JSON 输出不可直接解析 | 要求只输出 raw JSON，禁止额外解释和代码块 |

## 7. Agent Prompt 最小模板

```text
目标：

当前状态：

目标状态：

允许范围：

允许动作：

禁止动作：

执行步骤：

检查点：

停止并询问：

完成标准：
```

Agent prompt 必须写清边界。凡是删除文件、改依赖、改数据库 schema、发送消息、提交表单、购买、外部发布，都要加入人工确认。

Agent prompt 至少要回答：

- 从什么状态开始。
- 要到什么状态结束。
- 可以动哪些文件、页面、工具或资源。
- 明确不能做什么。
- 中途什么时候报告进展。
- 遇到什么动作必须先问用户。
- 什么结果算完成。

## 8. JSON 抽取 Prompt 最小模板

```text
你是一个信息抽取助手。你的任务是只基于输入内容抽取结构化 JSON。

约束：
- 只输出一个合法 JSON 对象。
- JSON 外不要输出任何解释、Markdown 或代码块。
- 所有字段必须来自输入，除非字段说明允许判断。
- 无法识别的字段返回 null。
- 枚举字段只能使用给定枚举，无法判断返回 null。

字段：
- field_name: 类型，来源，缺失值策略，枚举值（如有）

输入：
<<<
[用户输入]
>>>

输出格式：
{
  "field_name": null
}
```

## 9. 5 项质量检查

交付前做 5 项质量检查：

| 维度 | 检查问题 |
|---|---|
| 清晰度 | 任务动作是否明确？ |
| 具体性 | 约束、字段、格式是否具体？ |
| 上下文 | 模型是否需要靠猜背景？ |
| 完整性 | 输入、输出、完成标准是否齐全？ |
| 结构性 | 指令、输入、样例、输出是否分区？ |

不需要给用户展示评分。这个检查只用于交付前修正 prompt。

## 10. 最终自检

交付前只做这几个检查：

- prompt 是否能直接复制使用。
- 任务是否是具体动作。
- 输入和输出边界是否清楚。
- 输出形态是否唯一。
- 字段、枚举、缺失值、来源约束是否完整。
- Agent 是否有范围、禁止动作、检查点和停止条件。
- 有没有密钥、token、cookie、账号信息。
- 有没有为了显得高级而套了不必要框架。
- 有没有不改变输出的废话、客套、泛泛强调或重复限制。
