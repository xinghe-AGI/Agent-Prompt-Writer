# AI Prompt Writer

## 什么时候使用

当用户明确要求写 prompt、优化 prompt、改写 prompt、诊断 prompt，或把某个需求整理成可复制的 AI 工具提示词时，使用这个 skill。

不要因为用户只是要写文章、改代码、总结资料、生成方案，就自动把任务转成 prompt。只有用户要的是“提示词本身”时才触发。

如果用户只是给出普通任务，直接完成任务；如果用户说“帮我写给某个 AI 工具用的提示词”，才进入 prompt 写作流程。

## 核心原则

这个 skill 以结构化 Prompt 写法为主，不追求复杂框架。优先让 prompt 满足四个标准：

- **可解析**：输出形态明确，下游能稳定读取。
- **可复用**：同一骨架可以换字段、换场景继续用。
- **可维护**：版本、字段、规则、样例能同步更新。
- **可迭代**：改动后能知道改了什么、为什么改。

完整方法见 `references/prompt-method.md`。

## 工作流程

1. **判断用途**
   - 一次性任务：写短 prompt。
   - 结构化输出 / JSON / 自动化工作流：使用完整结构化骨架。
   - 长期复用角色：加入简介、版本、初始化和交互规则。
   - Agent 执行任务：写清范围、禁止动作、检查点和停止条件。

2. **补齐关键信息**
   - 输入是什么。
   - 输出是什么。
   - 字段有哪些。
   - 哪些字段必须来自输入。
   - 缺失值怎么处理。
   - 异常输入怎么处理。
   - 什么结果算完成。
   - 如果信息不足，最多问 3 个问题。优先问目标工具、输出形态、字段/范围/异常处理。

3. **选择最小结构**
   - 简单需求只保留：角色、任务、约束、输出格式。
   - 生产级需求使用：角色、背景、简介、任务、约束条件、技能、工作流、输出格式、输入模板、输出样例、初始化。

4. **生成可复制 prompt**
   - 不解释理论，除非用户要求。
   - 不输出多个版本，除非用户要求比较。
   - 不写入密钥、API key、token、cookie、连接字符串或账号凭据，统一用 `[ENV_VAR_NAME]` 占位。

5. **交付前自检**
   - 输出形态是否只有一个。
   - 字段契约是否前后一致。
   - 缺失值策略是否可执行。
   - 枚举值是否可校验。
   - 来源约束是否能防止幻觉。
   - 规则是否写成“条件 + 结果”。
   - 样例是否覆盖正常和异常情况。
   - 是否存在模糊任务、多任务混杂、输入边界不清、缺少成功标准等坏 prompt 模式。
   - 是否删掉了不改变输出的客套话、空泛强调和重复限制。

## 默认输出格式

默认只输出一个可复制 prompt：

```text
[这里放可直接复制的 prompt]
```

然后用一行说明：

`用途：结构化输出 / Agent 执行 / 角色提示词 / 一次性任务`

如果用户给出的信息不足以决定字段、输出形态或目标工具，最多先问 3 个问题。

## 写作偏好

- 直接、清楚、少废话。
- 每句话都必须改变预期输出；不改变输出的句子删掉。
- 少用花哨框架名，除非用户要学习框架。
- 不机械套用 26 条提示词技巧，只选择能减少歧义的几条。
- 不默认使用“小费激励”“惩罚语句”。
- 对推理原生模型，不默认要求展示完整思维链。
