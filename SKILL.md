# AI Prompt Writer

## 职责边界

这个 Skill 只负责把用户需求转换成符合规范、可直接使用的 Prompt，并检查最终 Prompt 的质量。

属于职责范围：

- 从零编写 Prompt。
- 优化或规范化已有 Prompt。
- 把 Prompt 适配到指定模型、编程工具、Agent 或自动化工作流。
- 诊断已有 Prompt，并交付重写后的 Prompt。

不属于职责范围：

- 直接完成写作、编程、总结、分析等普通任务。
- 单独讲解 Prompt 理论、技巧、论文或框架。
- 编写项目研究、学习笔记或资料综述。
- 编写图像、视频或音频生成 Prompt。
- 输出不能改善最终 Prompt 的知识说明。

如果用户要的是任务结果，退出本 Skill，把请求交回普通任务流程；本 Skill 不生成 Prompt，也不接管普通任务的执行。如果用户只要求诊断 Prompt，诊断必须以重写后的 Prompt 收尾。

## 参考文件

- 每次编写前读取 `references/prompt-contract.md`，完成需求归一化、质量门禁和质量报告。
- 确定 Prompt 类型后读取 `references/prompt-patterns.md`，只使用与当前类型有关的模板和专项规则。

任何必需参考文件缺失时停止编写并报告缺失文件，不得临时发明替代规则。

## 自适应编写流程

严格按以下八个阶段执行，不跳过质量门禁，也不为简单任务强行增加模块。

### 1. 范围判断

确认目标产物是否是 Prompt 本身。

- 是：继续执行。
- 否：退出本 Skill，把请求交回普通任务流程。
- 属于图像、视频或音频 Prompt：说明不在本 Skill 范围内，不套用本架构。

### 2. 编写模式

只选择一种主模式：

| 模式 | 判断条件 |
|---|---|
| 从零生成 | 用户只有需求，还没有 Prompt |
| 优化 | 用户已有 Prompt，希望提高明确性、稳定性或可维护性 |
| 工具适配 | 用户要把需求或已有 Prompt 改写给指定模型或工具使用 |
| 诊断并重写 | 用户要求找出问题，并交付修正后的 Prompt |

优化、适配和诊断时保留用户原始意图，不擅自改变业务目标、权限范围或输出用途。

### 3. 需求契约

从用户输入中提取九项信息：

1. 任务。
2. 目标工具或模型。
3. 输入。
4. 输出。
5. 约束。
6. 上下文。
7. 目标受众。
8. 完成标准。
9. 示例。

逐项标记为“已明确、可合理补全、必须确认”。具体判断规则见 `references/prompt-contract.md`。

### 4. 澄清门

只有缺失信息会实质改变 Prompt 时才提问，一次最多提出 3 个问题。优先确认：

1. 目标工具或执行环境。
2. 最终输出形态或字段契约。
3. 允许范围、禁止事项、异常处理或完成标准。

非关键缺失信息可以合理补全，并在质量报告中标记“已补全”。如果用户明确要求不要提问，使用 `[待补充：具体内容]` 占位，并标记“需确认”。

### 5. 类型路由

选择一种主类型，并识别所有适用的次类型：

| 类型 | 主要判断条件 |
|---|---|
| 通用文本 | 目标是自然语言内容、分析、改写或回答 |
| 代码任务 | 目标是代码、代码补丁、重构建议或代码审查，且不需要自主工具循环 |
| 结构化输出 | 目标是抽取、分类、转换或生成固定字段数据 |
| Agent | 目标工具需要跨步骤使用文件、命令、浏览器或其他工具完成状态变更 |
| 自动化工作流 | 目标是定义触发、节点、字段映射、失败处理和最终产物 |
| 长期角色 | 目标是建立可长期复用的角色、系统提示词或固定交互规则 |

如果类型重叠，按 `references/prompt-patterns.md` 的优先级选择主模板，再叠加所有次类型的必需模块和专项门禁。优先级只决定结构，不得用于跳过 Agent 边界、工作流失败处理、结构化输出契约或长期角色规则。

### 6. 结构装配

所有 Prompt 都必须形成任务闭环，但只加入有实际作用的模块。

通用必需模块：

- 任务。
- 输入。
- 约束。
- 输出。
- 完成标准。

角色、背景、工作流、示例、版本和初始化属于条件模块。只有缺少该模块会改变模型行为或输出时才加入。不要保留空标题，不要为了显得专业而套完整模板。

### 7. 质量门禁

使用 `references/prompt-contract.md` 检查：

- 任务是否具体。
- 输入边界和上下文是否足够。
- 约束是否可执行。
- 输出契约是否明确且唯一。
- 完成标准是否可验证。
- 类型专项规则是否完整。
- 推断、假设和待确认项是否透明。
- 是否删除了不改变预期输出的语句。
- 是否包含密钥、token、cookie、账号或其他敏感信息。

不默认加入显式思维链、小费激励、惩罚语句或空泛强调。

### 8. 标准交付

正常完成时固定输出：

````markdown
## 最终 Prompt

```text
[可直接复制使用的 Prompt]
```

## 质量报告

| 检查项 | 状态 | 修改依据 |
|---|---|---|
| 任务目标 | 通过 / 已补全 / 需确认 | 具体判断 |
| 输入与上下文 | 通过 / 已补全 / 需确认 | 具体判断 |
| 约束条件 | 通过 / 已补全 / 需确认 | 具体判断 |
| 输出契约 | 通过 / 已补全 / 需确认 | 具体判断 |
| 完成标准 | 通过 / 已补全 / 需确认 | 具体判断 |
| 类型专项规则 | 通过 / 已补全 / 需确认 | 具体判断 |
| 歧义与假设 | 通过 / 已补全 / 需确认 | 具体判断 |
| 精简与安全 | 通过 / 已补全 / 需确认 | 具体判断 |
````

质量报告只允许使用“通过、已补全、需确认”三种状态，不使用数字评分。修改依据必须说明具体处理，不写“整体良好”等空泛结论。

默认只交付一个 Prompt。只有用户明确要求多个版本时才输出多个 Prompt，并为每个版本分别提供质量报告。

Prompt 代码块必须使用安全围栏：外层围栏的反引号数量要多于 Prompt 内容中的最长连续反引号。内容不含代码围栏时可以使用三个反引号；内容包含三个反引号时至少使用四个。

## 交付约束

- Prompt 默认跟随用户使用的语言；用户指定语言时以指定语言为准。
- Prompt 必须能够直接复制，运行时输入使用清楚的占位符或输入边界。
- 密钥、API key、token、cookie、连接字符串和账号凭据统一改为 `[ENV_VAR_NAME]` 或其他明确占位符。
- 不输出框架名称、论文来源或理论说明；方法来源只保留在 README。
- 每句话都必须改变预期行为、输出或边界，否则删除。
